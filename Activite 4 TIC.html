    1) Réseau (informatique) Network :

    • Un réseau informatique est un ensemble de dispositifs connectés les uns aux autres pour permettre l'échange d'informations numériques dans un domaine géographique donné, en respectant des règles prédéfinies.
    • Ce domaine peut être aussi petit que deux ordinateurs connectés ensemble ou aussi grand qu'un réseau mondial tel que l'Internet.


    La mise en réseau (Networking) :

    consiste à connecter des ordinateurs et des équipements électroniques entre eux, afin qu'ils puissent communiquer et partager des ressources. Cette connexion permet à différents programmes et applications de partager des données et des informations, quels que soient la distance et la localisation des différents équipements. Ainsi, un réseau est un ensemble de dispositifs électroniques interconnectés pour permettre la communication de données entre applications.


    Evolution :

    Un réseau informatique est un ensemble de dispositifs électroniques connectés pour permettre la communication et l'échange de données, à la fois entre des terminaux distants et entre des ordinateurs. À l'origine, les réseaux étaient destinés uniquement au transport de données informatiques, mais ils ont évolué pour intégrer également la voix et la vidéo. Les réseaux modernes permettent donc de communiquer non seulement par écrit, mais aussi de manière orale et visuelle.


    Le réseau informatique repose sur deux notions fondamentales :
    l'interconnexion, qui permet la transmission des données entre les différents nœuds, et la communication, qui permet l'échange de données entre les programmes en cours d'exécution. Chaque nœud représente l'extrémité d'une connexion, et chaque programme en cours d'exécution représente le bout d'une communication dans le réseau informatique.

    2)Le Modèle OSI



    OSI (Open Systems Interconnection) est un modèle de référence pour les communications de réseau qui a 7 couches distincts:


    Couche

    Fonctionement
    physique La transmission des données sur le support physique
    liaison de données La transmission des données entre les appareils qui partagent un même support physique.
    réseau Le routage des données à travers le réseau.
    transport L'acheminement des données de bout en bout entre les appareil
    session L'établissement et de la gestion des sessions entre les appareils
    présentation La conversion des données dans un format qui peut être compris par l'appareil de destination.
    application La communication entre les applications sur les appareils de réseau

    3)Le Modèle TCP/IP

    Couche

    Fonctionement

    réseau Le routage des données à travers le réseau
    transport l'acheminement des données de bout en bout entre les appareils
    application la communication entre les applications sur les appareils de réseau.

    Chapitre 5: Système de Gestion de Version

    1)Qu'est-ce que Git ?

    Git est un logiciel de versioning créé par Linus Torvalds en 2005, qui permet de conserver un historique des modifications effectuées sur un projet afin de pouvoir y revenir en cas de problème. Les logiciels de gestion de versions sont indispensables pour la gestion de projets et pour travailler en équipe de manière efficace. Parmi les logiciels de gestion de versions, Git est le leader incontesté et est donc indispensable pour tout développeur.

    2) A quoi sert concrètement un système de gestion de version ?

    la gestion de versions devient de plus en plus complexe lorsqu'on travaille en équipe sur un site web, car il faut s'assurer que les fonctionnalités sur lesquelles travaillent les différents développeurs ne rentrent pas en conflit et que chacun dispose d'une version à jour du site pour tester et implémenter ses fonctionnalités. Pour résoudre ces problèmes, un système de gestion de version décentralisé est mis en place, et Git est présenté comme un outil de coordination qui permet aux développeurs de travailler sur différentes versions d'un même fichier et d'accéder à l'historique des modifications pour savoir qui a fait quoi. Git offre également de nombreuses fonctionnalités supplémentaires pour synchroniser les dossiers locaux avec le serveur et mettre à jour les différences automatiquement.

    3) Les deux modèles des logiciels de gestion de version : modèle centralisé vs modèle décentralisé

    les logiciels de gestion de version sont basés sur deux modèles : le modèle centralisé et le modèle décentralisé. Dans le modèle centralisé, la source du code du projet est hébergée sur un serveur distant central, tandis que dans le modèle décentralisé, chaque utilisateur télécharge et héberge l'intégralité du code source du projet sur sa propre machine. Le modèle décentralisé est associé à Git et présente plusieurs avantages, notamment en termes de flexibilité et de sécurité. Les utilisateurs peuvent travailler en ligne sur leur propre machine sans être connectés à un serveur central, et chaque utilisateur possède une copie complète du projet qui peut être utilisée comme back-up en cas de corruption du serveur central.

    4) Qu'est-ce que GitHub ?

    Git est un logiciel de gestion de version qui permet d'enregistrer les différentes modifications effectuées sur un projet et de revenir à une version précédente si nécessaire. Dans le langage de la gestion de version, une copie de l'ensemble des fichiers d'un projet et de leur version située sur le serveur central est appelée un dépôt ou repository. GitHub est un service en ligne qui permet d'héberger des dépôts Git et est le plus grand hébergeur de dépôts Git du monde. La plupart des dépôts hébergés sur GitHub sont publics, ce qui signifie que tout le monde peut télécharger le code de ces dépôts et contribuer à leur développement en proposant de nouvelles fonctionnalités. En résumé, Git est un logiciel de gestion de version, tandis que GitHub est un service en ligne d'hébergement de dépôts Git qui fait office de serveur central pour ces dépôts.

    5) Utiliser Git : ligne de commande, console et interface graphique

    Le texte explique que dans ce cours, nous allons utiliser la console et donc un langage en lignes de commande pour installer et utiliser Git. Cette décision est prise pour différentes raisons : utiliser la ligne de commande est le seul moyen d'avoir accès à toutes les commandes Git, tout le monde aura accès aux mêmes commandes avec la ligne de commande, et si vous savez comment utiliser la version en ligne de commande, vous saurez utiliser n'importe quelle autre interface graphique. Pour ouvrir la ligne de commande sur Windows, il est possible de soit utiliser un raccourci clavier, soit rechercher "cmd" dans la barre de recherche.

    6) Installation de Git

    7) Paramétrage de Git

    pour éviter d'avoir à fournir à nouveau certaines données, telles que le nom d'utilisateur et l'adresse e-mail, il est possible de les enregistrer dans la configuration de Git en utilisant la commande git config. En passant l'option --global, ces informations seront utilisées globalement pour tous les projets Git. Pour renseigner un nom et une adresse e-mail, on peut utiliser les commandes suivantes : git config --global user.name "Bouchekir Redouane" et git config --global user.email redouane.bouchekir@gmail.com, en remplaçant les informations par ses propres données. Pour vérifier que les informations ont bien été enregistrées, il est possible de taper git config user.name et git config user.email, qui devraient renvoyer les informations entrées.

    8) Démarrer un dépôt Git

    Un “dépôt” correspond à la copie et à l'importation de l'ensemble des fichiers d'un projet dans Git. Il existe deux façons de créer un dépôt Git : • On peut importer un répertoire déjà existant dans Git ; • On peut cloner un dépôt Git déjà existant. Nous allons voir comment faire cela dans la suite de ce cours. Avant cela, je pense qu'il est bon de comprendre comment Git conçoit la gestion des informations ainsi que le fonctionnement général de Git.

    9) La gestion des informations selon Git

    Git considère les données à la manière d'un flux d'instantanés ou "snapshots". Chaque fois qu'un utilisateur enregistre l'état d'un projet dans Git, il prend un instantané du contenu de l'espace de travail à ce moment-là et enregistre une référence à cet instantané pour y accéder ultérieurement. Ces instantanés sont stockés localement dans une base de données, ce qui signifie que la plupart des opérations de Git peuvent être effectuées localement sans avoir besoin d'être connecté à un serveur central distant. Cette caractéristique rend les opérations plus rapides et le travail plus agréable en général.

    10) Les états des fichiers

    Git suit les modifications apportées aux fichiers d'un projet en utilisant deux grands états : suivi et non suivi. Un fichier est suivi s'il appartient au dernier instantané capturé par Git et est enregistré dans la base de données locale. Tout fichier qui n'appartient pas au dernier instantané et qui n'a pas été indexé est considéré comme non suivi. Chaque fichier suivi peut avoir l'un des trois états suivants : modifié, indexé ou validé. Lorsque l'on démarre un dépôt Git, les fichiers sont initialement non suivis et doivent être indexés et validés. Lorsque l'on travaille sur un projet, les fichiers modifiés sont considérés comme modifiés par Git jusqu'à ce qu'ils soient indexés. On indexe un fichier en indiquant à Git que le fichier modifié ou le nouveau fichier doit faire partie du prochain instantané. Lorsque l'on demande à Git de prendre un instantané, les fichiers faisant partie de l'instantané sont de nouveau considérés comme validés et le cycle peut recommencer.

    11) Les zones de travail

    Le texte explique que les états des fichiers sont liés à des zones de travail dans Git et que tout projet Git est composé de trois sections : le répertoire de travail, la zone d'index et le répertoire Git. Le répertoire de travail correspond à une extraction unique d'une version du projet où les fichiers sont extraits de la base de données compressée située dans le répertoire Git et sont placés sur le disque pour être utilisés ou modifiés. La zone d'index stocke les informations concernant ce qui fera partie du prochain instantané ou commit. Le répertoire Git est l'endroit où Git stocke les méta-données et la base de données des objets du projet. Le processus de travail consiste à travailler sur les fichiers dans le répertoire de travail, à indexer les fichiers modifiés ou créés, à valider la version indexée pour l'ajouter au répertoire Git. Des schémas sont fournis pour aider à visualiser ces concepts.

    12) Créer un dépôt Git à partir d'un répertoire existant

    lorsqu'on démarre avec Git, on peut importer un projet existant dans Git en utilisant la commande git init pour initialiser un dépôt Git. Cette commande crée un sous-répertoire .git qui contient des fichiers nécessaires au fonctionnement de Git. La commande git status permet de connaître l'état des fichiers dans le répertoire. Si des fichiers ne sont pas sous suivi de version, on peut utiliser la commande git add pour les indexer. Les fileglobs peuvent être utilisés pour ajouter des fichiers correspondant à un schéma de correspondance. La commande git commit permet de valider les fichiers indexés et de les ajouter en base avec un message de commit. Bien documenter chaque commit permet aux contributeurs de comprendre les modifications apportées. Après un commit, la commande git status indique que tous les fichiers sont sous suivi de version et qu'aucune modification n'a été apportée depuis le dernier commit.

    13) Cloner un dépôt Git

    une autre façon de démarrer un dépôt Git est de cloner localement un dépôt Git existant en utilisant la commande git clone. Pour cela, on doit d'abord créer un dépôt sur GitHub, puis on peut transmettre nos fichiers à ce dépôt en utilisant les commandes git branch -M main, git remote add origin et git push -u origin main. Si on vérifie le dépôt sur GitHub, on peut voir les fichiers qui ont été ajoutés.

    14) Ajouter ou modifier des fichiers dans un projet et actualiser notre dépôt Git

    en travaillant sur un projet, on sera amené à ajouter, modifier et supprimer des fichiers. On doit utiliser les commandes git add et git commit pour indiquer tous ces changements à Git afin qu'il conserve un historique des versions. Il est important de noter que le commit d'un fichier se basera sur son état au moment du git add. Si un fichier est modifié après le git add, le commit ne prendra pas en compte les dernières modifications. Pour enregistrer toujours la dernière version d'un fichier, il est donc conseillé d'effectuer un git add juste avant un git commit. On peut également utiliser la commande git commit -a pour mettre en un coup les fichiers modifiés et déjà sous suivi dans la zone d'index et les valider, sans avoir à taper git add.

    15) Consulter l'historique des modifications Git

    La manière la plus simple de consulter l'historique des modifications Git est d'utiliser la commande git log. Cette commande affiche la liste des commits réalisés du plus récent au plus ancien. Par défaut, chaque commit est affiché avec sa somme de contrôle SHA-1, le nom et l'e-mail de l'auteur, la date et le message du commit.

    16) Annuler des modifications apportées à un fichier

    l'un des principaux intérêts d'utiliser un logiciel de gestion de version est de pouvoir "rollback" ou revenir à un état antérieur enregistré d'un projet. Après un commit, on continue à travailler sur les fichiers et à les modifier. Parfois, certaines modifications ne fonctionnent pas comme prévu et on veut revenir à l'état du dernier instantané Git, c'est-à-dire au dernier état enregistré. On peut faire cela en utilisant la commande git checkout -- nom-du-fichier ou la nouvelle commande spécialisée git restore.

    17) Qu'est ce qu'une branche ?

    créer une branche dans Git revient à créer une "copie" du projet pour développer et tester de nouvelles fonctionnalités sans affecter le projet principal. Contrairement à la plupart des systèmes de contrôle de version, Git n'effectue pas de copie physique de l'ensemble du répertoire de travail pour créer une nouvelle branche, ce qui rend la création et la fusion de branches très facile. Dans Git, une branche est simplement un pointeur vers un commit et est représentée par un simple fichier contenant les 40 caractères de l'empreinte SHA-1 du commit sur lequel elle pointe. Lorsqu'on effectue un commit, Git stocke un objet commit qui contient des informations sur l'auteur du commit, le message et un pointeur vers l'instantané du contenu indexé et des pointeurs vers le ou les commits précédant directement le commit courant. La branche par défaut dans Git s'appelle master et est un pointeur qui se déplace automatiquement à chaque nouveau commit pour pointer sur le dernier commit effectué. En résumé, créer une nouvelle branche dans Git crée simplement un nouveau pointeur plutôt que de recopier l'intégralité du répertoire de travail.
